Segmento A — Contexto del Proyecto

¿Qué es TicketFlow?

API REST de gestión de soporte técnico — permite a usuarios crear tickets, adjuntar archivos, asignarlos a agentes y recibir notificaciones asíncronas. Backend Node.js/TypeScript sobre EC2 privada.

Red & Ingress: EC2 Privada + ALB

❌ Alternativa Descartada
🚨EC2 en subred pública con IP elástica
🚨Puerto 8080 expuesto directamente a Internet
🚨Sin separación entre capa pública y aplicación
VS
✅ Decisión Adoptada
🛡️EC2 en subred privada 10.0.11.0/24 — sin IP pública
🛡️ALB en subnets públicas como único punto de entrada (puerto 80)
🛡️App Security Group: solo acepta tráfico del SG del ALB

Diseño de red — VPC 10.0.0.0/16

# Subnets públicas (ALB + NAT Gateway) Public A: 10.0.1.0/24 us-west-2a Public B: 10.0.2.0/24 us-west-2b # Subnets privadas (EC2 Application) Private A: 10.0.11.0/24 us-west-2a ← EC2 Private B: 10.0.12.0/24 us-west-2b (crecimiento)

NAT Gateway vs VPC Endpoints

✅NAT Gateway: elegido por simplicidad y flexibilidad — permite instalar dependencias, acceder APIs externas y hacer bootstrap
⚠️VPC Endpoints descartados: no permiten salida a Internet para bootstrap de EC2 ni consumo de APIs externas
📝Para stageucción real: complementar con Gateway Endpoints para S3 y DynamoDB reduciendo tráfico y costo

Segmento B · Highlight #2 — Entrega 4

Pipeline CI/CD con Gate de Aprobación

El workflow terraform-ci.yml orquesta 6 jobs encadenados. La decisión clave: separar el plan del apply y exigir aprobación explícita antes del deploy a stagingucción.

terraform fmt
→
terraform validate
→
terraform plan (dev)
→
dev apply env: dev
→
validate stage plan
→
stage apply 🔐 gate

Decisión: Plan separado del Apply

✅Plan en PR: genera tfplan-dev y lo sube como artefacto — el apply descarga el plan exacto aprobado
✅Sin re-planificación: elimina la posibilidad de que el apply ejecute cambios distintos a los revisados
❌Alternativa descartada: plan + apply en un solo job haría imposible auditar qué se aprobó

OIDC — Sin credenciales estáticas

- name: Configure AWS Credentials uses: aws-actions/configure-aws-credentials@v4 with: role-to-assume: ${{ vars.AWS_CI_ROLE_ARN }} aws-region: us-west-2 # Sujeto OIDC restringido a main: # repo:solivalle/..:ref:refs/heads/main
🔐 Environment Gate El job staging_apply usa environment: staging — GitHub pausa la ejecución y requiere aprobación manual de un revisor autorizado antes de desplegar.
Segmento B · Highlight #3 — Entrega 5

Seguridad: OIDC + KMS + Secrets Manager

🔑 OIDC Federation
✅CI Runner Role asume credenciales temporales via token OIDC de GitHub Actions
✅Trust policy scoped a refs/heads/main — PRs no obtienen rol de deploy
❌Credenciales AWS_ACCESS_KEY_ID y SECRET eliminadas del repositorio
🗝️ KMS — CMK por entorno
✅Una CMK por entorno (alias/ticketflow-dev-cmk)
✅Cifra: S3 buckets · DynamoDB · Secrets Manager
⚖️CMK compartida vs por-servicio: elegida compartida para reducir overhead operacional en este proyecto
🔒 Secrets Manager
✅JWT_SECRET almacenado en AWS Secrets Manager — Terraform solo inyecta el ARN a EC2
✅Bootstrap: EC2 obtiene el secreto en arranque y lo escribe en /etc/ticketflow.env con permisos 0600
❌Patrón eliminado: variables plaintext en workflow YAML

Segmento C — Reflexión & Trade-off

Guardrails y Decisiones Arquitectónicas

🛑 Guardrail: Branch Ruleset de GitHub
🔐Mecanismo: Branch protection ruleset en main — requiere Pull Request antes de hacer merge; pushes directos son rechazados
⚙️Required checks: terraform fmt, terraform validate, terraform plan deben pasar — si fallan, el merge está bloqueado
🎯Cómo atrapa errores: Un archivo mal formateado o con error de sintaxis en Terraform provoca que fmt o validate fallen y el PR queda bloqueado automáticamente
🔐 Guardrail: Environment Approval Gate
⚙️Mecanismo: Job staging_apply con environment: staging — GitHub pausa el workflow y espera aprobación explícita del revisor requerido
🎯Sin gate: cualquier push a main desplegaría automáticamente a stageucción sin revisión humana
⚖️ Trade-off Arquitectónico: EC2 vs Lambda
✅EC2 elegida: la aplicación mantiene estado de conexiones WebSocket para notificaciones en tiempo real — Lambda con cold starts rompería la experiencia
❌Lambda descartada: cold starts en tráfico esporádico + imposibilidad de mantener conexiones WebSocket persistentes
⚠️Costo asumido: costo fijo de EC2 t4g.nano mitigado por arquitectura ARM64 (~$3.5/mes)
⚖️ Trade-off: CMK compartida vs por-servicio
✅CMK compartida: una clave por entorno con key policy restrictiva — cifra S3, DynamoDB y Secrets Manager con menor overhead operacional
❌Por-servicio descartado: 3+ claves = gestión de políticas multiplicada sin ganancia proporcional para este proyecto

Segmento D — Cambio en Vivo: Implementación

El Instructor Asigna — El Equipo Implementa

🔄 Secuencia de ejecución

1️⃣Instructor asigna el comportamiento exacto a implementar
2️⃣Crear rama: git checkout -b feature/live-change
3️⃣Implementar con asistencia de IA en el handler correcto
4️⃣Commit + push: git push origin feature/live-change
5️⃣Abrir Pull Request apuntando a main
6️⃣CI pipeline arranca — Parte 1 concluye aquí

⏱️ Interludio del Pipeline

✅terraform fmt — verifica formato de todos los archivos .tf
✅terraform validate — valida sintaxis y referencias
✅terraform plan — genera plan contra estado dev en S3
🔍PR review — aprobación requerida por CODEOWNER
🚀Merge → auto-deploy a entorno dev

📋 Líneas base (antes del cambio)

Antes de implementar: ejecutar curl -H "Authorization: Bearer $TOKEN" $BASE_URL/api/tickets y capturar la respuesta actual. Esta línea base es el "antes" con el que se contrastará el resultado en el Segmento E. Sin línea base demostrada, no hay prueba del cambio.

Durante el gate de staging

2️⃣Confirmar que el dev deploy completó exitosamente
3️⃣Un compañero aprueba el gate de staging en vivo durante la presentación
4️⃣Narrar cada paso mientras el pipeline despliega el nuevo image
⚠️ Si el pipeline va en rojo
📖Leer el log de error — diagnosticar el problema en voz alta
🔧Aplicar fix en la misma rama o hacer rollback
❌Nunca: ignorar el error, pedir que se omita, deshabilitar el check

Verificación del cambio — Antes vs Después

# ANTES del cambio: curl -H "Authorization: Bearer $TOKEN" \ $STAGING_URL/api/tickets # → Respuesta SIN el nuevo campo # DESPUÉS del deploy: curl -H "Authorization: Bearer $TOKEN" \ $STAGING_URL/api/tickets # → Respuesta CON el nuevo comportamiento # → Contraste demostrado en vivo 🎯
Q&A — Comprensión Conceptual del Equipo

Preguntas que Todo Miembro Debe Responder

Branch Ruleset
❓¿Por qué el ruleset impide el merge cuando un check falla pero un required check solo no lo hace?
💡R: El ruleset define quién puede hacer merge y bajo qué condiciones — los required checks son la condición técnica, el ruleset es el mecanismo de enforcement
OIDC Trust Policy
❓¿Por qué el OIDC está restringido a refs/heads/main?
💡R: Solo main debe desplegar infraestructura — PRs y ramas de feature corren validación local sin asumir el rol de CI, evitando deploys no controlados
Terraform Plan
❓¿Por qué el plan se ejecuta contra el estado de staging y no dev?
💡R: El workspace stage tiene su propio estado remoto en S3 — planear contra dev sería planear contra infraestructura diferente
Environment Gate
❓¿Qué pasaría si removemos el gate de aprobación de staging?
💡R: Todo merge a main desplegaría automáticamente a stageucción sin revisión humana — el gate es la barrera de control manual
Handler de la Aplicación
❓¿Por qué el handler asignado es el lugar correcto para el cambio?
💡R: La capa de rutas → service → data repository es responsabilidad separada — el cambio observable en la respuesta vive en la capa de servicio o en el mapper de respuesta
Código generado por IA
❓¿Qué hace exactamente la función que generó la IA?
💡R: Todo miembro debe poder explicar línea por línea el código generado — el asistente escribe, el equipo comprende y defiende
